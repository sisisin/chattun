### 指摘事項

1. **[front/src/app/features/mrkdwn/parser.ts:17-24]**: 閉じていないコードブロック（` ``` ` が1つだけ出現）の処理が壊れている。` ```code ` のように閉じる ` ``` ` がない場合、コードブロック処理が `endIndex=-1` でスキップされた後、インラインコード処理 (`input[pos] === '``'`) にフォールスルーし、2番目のバックティックをインラインコードの閉じマーカーとして誤認識する。結果として `{type:'code', text:'`'}` と `{type:'text', text:'code'}` という誤ったノードが生成される。Slackの実メッセージでは閉じていないコードブロックが送られることがあるため、フォールスルーしてプレーンテキストとして扱うのが正しい。

   修正案: ` ``` ` の処理がスキップされた場合（`endIndex === -1`）は、その位置の文字をプレーンテキストとして蓄積するか、もしくは ` ``` ` を1つのテキストとして出力してから `pos += 3` して次のループへ進む。

2. **[front/src/app/features/mrkdwn/parser.ts:101-103]**: `flushText` 関数が完全なデッドコードになっている。本体が `// No-op` コメントのみで、呼び出し元（20行目）も含めて何の機能も果たしていない。コードブロック検出前のテキスト蓄積はメインループで別途処理されており、この関数は不要。関数定義と呼び出しをまとめて削除すべき。

3. **[front/src/app/features/mrkdwn/parser.ts:93-98]**: `findClosing` の `i > start` という条件が、インラインコード処理で意図どおりに動作するが、条件の意味がコメントで説明されていない。`start = pos + 1` で呼び出されているため、`i === start` の場合（つまり開きマーカーの直後の文字がマーカーと同じ）は閉じマーカーとして認識しない。この条件が「空のインラインコードを拒否する」ためか「開きバックティック直後のバックティックをスキップする」ためなのかをコメントで明示することが望ましい。

4. **[front/src/app/features/mrkdwn/parser.ts:118-124]**: `isSpecialAt` 関数内の ` if (input.startsWith('```', pos)) return true; ` は冗長。直前の `if (ch === '\n' || ch === '``') return true;` で `ch === '``'` が既に true になるため、` ``` ` の先頭文字のケースでもその条件が先にヒットする。削除してよい（軽微）。
