### 指摘事項

1. **[server/src/io.ts:19]**: `socket.emit('auth_error', ...)` の直後に `socket.disconnect()` を呼んでいるが、Socket.IOのemitはバッファリングされるためdisconnect前にパケットが送信される保証はない。Socket.IOでは`socket.emit`の後に`socket.disconnect()`を即時呼び出すと、emitしたイベントがフラッシュされる前に接続が切断される可能性がある。`disconnect()`の第一引数にオプションを渡すか、もしくは`socket.emit`にackコールバックを設けてackを受け取ってからdisconnectするか、あるいは`socket.disconnect(true)`ではなく`socket.emit`後に短いディレイを置く実装を検討すること。最もシンプルな対応としては、`socket.emit`の代わりに`socket.disconnect()`のみとし、クライアント側でdisconnectイベントの`reason`を見て判断する方式に変更するか、またはSocket.IOの`io.disconnectSockets()`で`close`フラグを使う方法を検討する。

2. **[front/src/app/services/rtm-socket.ts:16-20]**: `auth_error`ハンドラ内で`alert()`を使用しているが、`alert()`はブラウザの同期UIブロックを発生させるためUXとして好ましくない。既存の認証フロー（`authRequiredRoutesTransitionStarted`→`GET /connection`→401時の処理）がどのようなUI通知手段を使っているかを確認し、統一した通知方法を採用すること。既存コードで`alert()`が使われている実績があるなら許容範囲だが、そうでなければトースト等のノンブロッキングな通知UIへの変更を検討すること。

3. **[front/src/app/services/rtm-socket.ts:16]**: クライアント側の`auth_error`ハンドラはイベントの引数（`{ reason: 'session_expired' }`）を受け取っていない。サーバーが送る`reason`フィールドを将来的に使い分けることを想定しているなら、引数を受け取る形にしておくべき。現状では`reason`が無視されており、サーバー側でわざわざ`reason`を付与している意味がない。引数が不要なら、サーバー側もペイロードなし（`socket.emit('auth_error')`）にそろえることを検討すること。
