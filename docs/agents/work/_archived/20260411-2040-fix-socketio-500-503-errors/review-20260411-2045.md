## Round 1

STATUS: NEEDS_FIX

### 指摘事項

1. **[server/src/index.ts:177-184]**: `unhandledRejection`ハンドラが`reason`に対して`logger.errore`の第2引数として渡しているが、`reason`が`Error`でも`ErrorWithLogContext`でもない場合（例: 文字列やnullなど）、`else`ブランチで`logger.errore('Unhandled rejection', reason)`を呼ぶことになる。`errore`の第2引数の型は`unknown`であり`errorToLogObject`内で適切に処理されるため動作上の問題はないが、`reason`が`null`の場合`String(null) = "null"`となり情報が失われる。これは現状でも許容範囲だが、念のため`reason`が`null`/`undefined`の場合のログが分かりやすくなるよう考慮することを推奨する。（必須ではない）

2. **[server/src/index.ts:177-184]**: `unhandledRejection`ハンドラ内でプロセスを終了させていない。Node.jsのデフォルト動作（v15以降）は`unhandledRejection`でプロセスをクラッシュさせるものだが、本変更ではあえてクラッシュを防止することが目的であり、これは意図的な判断と読み取れる。ただし、**意図しない非同期エラーが無音で握り潰されるリスク**がある。たとえば、Socket.IOや他のライブラリが非同期エラーをthrowした場合、ログが残るだけでプロセスは継続し続ける。これが「500/503を防ぐために致命的でないエラーを無視する」という設計判断であれば問題ないが、致命的なエラー（例: DB接続の喪失、認証トークンの異常など）も黙って無視される可能性がある。コメントまたはドキュメントで「このハンドラはプロセスを終了しない意図的な設計」であることを明記することを推奨する。

3. **[server/src/index.ts:185-188]**: `uncaughtException`ハンドラでは`process.exit(1)`を呼んでいる。これは同期例外（`throw`されたエラー）を捕捉するもので、`unhandledRejection`（非同期のPromise rejection）とは別物である。このプロセス終了は適切だが、`uncaughtException`発生時には既にアプリケーション状態が不整合になっている可能性があり、`logger.errore`が正常に動作するかどうかも保証できない（例: `AsyncLocalStorage`の状態が壊れているケース）。現実的には大きな問題になりにくいが、認識しておくべき点である。（情報提供のみ、修正必須ではない）

4. **[server/src/index.ts:177-184]**: `unhandledRejection`ハンドラが`main()`関数の内側に登録されている。これはサーバー起動後にのみ登録されるが、`main()`の実行中（起動処理中）に発生したunhandled rejectionはこのハンドラでは捕捉されない。起動処理中のエラーは`main().catch`で捕捉されるため実害はないが、ハンドラをトップレベルに移動した方が意図が明確になる。修正を推奨する。

### Dismissed

- **指摘1**: `reason`が`null`/`undefined`の場合、`errorToLogObject`で`String(null)`として処理される。"null"という文字列がログに残るため情報としては十分で、追加のハンドリングは不要。
- **指摘3**: 情報提供のみの指摘。`uncaughtException`時の`logger.errore`の動作保証は難しいが、ベストエフォートで構造化ログに記録する方がstderrへの`console.error`よりも運用上有益。
