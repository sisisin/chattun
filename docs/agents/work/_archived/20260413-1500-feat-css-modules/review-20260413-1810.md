3. **[front/src/app/features/setting/components/ChannelMatchSetting.tsx:94,103]**: `settingStyles.settingGroup` への変更は行われているが、内部の `setting-group-title`、`setting-group-match`、`select-group`、`select-group-item`、`select-group-chaticon`、`button-remove`、`setting-group-footer`、`button-add`、`button-primary` は依然としてグローバルクラスのままになっている。同様に `DeepLinkSetting.tsx`、`MuteUsersSetting.tsx` も同じ状況。タスクの「各コンポーネントで `className` が CSS Modules のオブジェクトプロパティアクセスに変更されている」という達成条件に対して、これらは `Setting.module.css` に定義されていないクラスであるため、グローバルCSSとして `_base.css` や `_form.css` 等に定義されているものであれば問題ないが、その確認が必要。意図的にグローバルのまま残しているクラスと、CSS Modules 化すべきクラスの区別を明確にすること。

4. **[front/src/app/features/timeline/components/TimelineView.module.css:6]**: CSS Modules に変換後、クラス名が `.tweetlist.is-empty`（複合セレクタ）から `.tweetlist-empty`（単一クラス）に変更されている。対応して `TimelineView.tsx` と `ThreadView.tsx` では `styles.tweetlistEmpty` で参照されており実装は一致しているが、元の `.tweetlist.is-empty` はJSから `is-empty` を動的に付与できる設計だったのに対し、今回の変更で `tweetlist` と `tweetlistEmpty` が完全に分離されたことで意味的に変化している。これは正しい変更だが、元のHTMLの `className="tweetlist is-empty"` が `className={styles.tweetlistEmpty}` に単純化されており、`tweetlist` クラスのスタイル（`padding: 0; list-style: none`）が空の場合に適用されなくなっている。空リスト時に `tweetlist` のスタイルが必要かどうかを確認すること（現状では `ul.tweetlist` のスタイルは空リスト時に適用されていない）。
