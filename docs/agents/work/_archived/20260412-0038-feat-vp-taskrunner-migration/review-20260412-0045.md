2. **[front/package.json:18-19]**: `start` と `start-nm` の2スクリプトは `vp run --parallel` を使用しているため、vp task config の `dependsOn` では並列実行を表現できないと判断されてnpm scriptsに残されたと見られる。これはタスクの達成条件2（「vp run --parallel相当をconfigで表現できるか調査結果が明確」）に関わる部分だが、調査結果の記録（task.mdやDONE.md等）がなく、なぜconfigに移行できないかの根拠が示されていない。`dependsOn` は直列依存の表現であり、並列起動には対応していないことが理由と推測されるが、明示的に記録・コメントがあることが望ましい。

3. **[CLAUDE.md:56]**: CLAUDE.mdの更新後コマンド表記に誤りがある。`yarn vp test run` と記載されているが、正しくは `yarn vp run test-all` または `vp test run`（直接vpコマンドを使う場合）が適切。`vp test run` はVitest実行のCLIコマンドであり、`yarn vp test run` はyarn経由でvpを呼ぶ表記として正しいが、他のコマンドとの統一感で言えば `yarn vp run test-all`（run.tasksで定義したtest-allタスクを使う表記）の方がwayとしては一貫している。一方、`yarn vp lint` や `yarn vp lint --fix` はrun.tasksではなくvpのサブコマンドをそのまま呼んでおり、CLAUDE.mdにはrun.tasks経由と直接vpサブコマンド経由が混在している。利用者の混乱を避けるため、コマンドの由来（run.tasksのタスク名か、vpのサブコマンドか）を統一またはコメントで補足することを推奨する。

4. **[.github/workflows/pull--check.yaml:39]**: CI の `yarn vp run test-all` は `vp fmt --check && vp lint && vp test run` を実行する。format-checkがCIに含まれているのは意図的と見られ正しい。問題はないが、`test-all` タスクの実行はformat checkも含むことをCI側のコメントや step名で明示すると可読性が上がる（必須ではない）。
