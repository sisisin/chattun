## Round 1

STATUS: NEEDS_FIX

### 指摘事項

1. **[server/src/index.ts:110-113]**: `/api/connection`エンドポイントが引き続き`accessToken`をフロントエンドに返却している。プロキシ化によりフロントエンドがSlack APIトークンを保持する必要はなくなったにもかかわらず、`SessionState.accessToken`へのセット処理（`front/src/app/features/session/module.ts:39`）を含め、フロントで`accessToken`を扱うコードが残存している。アクセストークンをフロントエンドに渡し続けることはセキュリティ上の懸念となる。`/api/connection`から`accessToken`を除外し、フロントの`SessionState.accessToken`フィールド・`connectionInitialized`ペイロード・関連コードも合わせて削除することを推奨する。

2. **[server/src/index.ts:127-128]**: `express.json()`ミドルウェアがSlackプロキシルート定義の直前（127行目付近）に配置されており、アプリ全体のグローバルミドルウェアとして登録されていない。現状では既存のPOSTルートがないため実害はないが、将来的にPOSTルートを前方に追加した際に`req.body`が`undefined`になるバグを招く。`app.use(sessionMiddleware)`などと同じ位置（早期）にグローバル登録すべき。

3. **[front/src/app/services/http/SlackClient.ts:82-85]**: `startRtm`メソッドが`Promise.resolve({ ok: true } as any)`を返すスタブとして残っている。`ISlackClient`インターフェースおよび`SlackClient`クラスから`startRtm`メソッドを削除し、インターフェースの型整合性を保つことを推奨する。（呼び出し元がなければデッドコードとなる）

4. **[server/src/index.ts:130-133]**: `getSlackClient`はリクエストごとに毎回`new WebClient(accessToken)`を生成している。シングルユーザーアプリの実態を考えると大きな問題ではないが、軽量化のためにリクエスト単位での新規インスタンス生成ではなく、セッション由来のトークンに基づいたキャッシュや、`WebClient`を一度だけ生成してトークンを差し替えるパターンを検討する価値がある。（必須ではない改善提案）

### Dismissed

- **指摘4**: マルチユーザー対応のアプリであり、各リクエストのセッションから異なるアクセストークンを取得する設計上、リクエストごとにWebClientを生成するのは妥当。トークンキャッシュはセッション間のトークン混在リスクを生む。WebClientインスタンスは軽量であり、パフォーマンス上の問題はない。
