## Round 1

### 指摘事項

1. **[front/src/app/features/mrkdwn/parser.ts:38] 単語中間のアンダースコアがitalicにパースされるバグ**: `_([^_\n]+)_` の正規表現は単語境界を考慮していないため、`foo_bar_baz` のような識別子内のアンダースコアも `italic("bar")` としてパースされる。Slackの実際の仕様では単語の境界外にあるアンダースコア（直前後が英数字）はitalicとして扱われない。同様に `*` も `2 * 3 * 4` のような場合に `bold(" 3 ")` としてパースされてしまう。

   修正案: 正規表現にword boundaryの考慮を加える。例として `_` の場合、`(?<![a-zA-Z0-9])_([^_\n]+)_(?![a-zA-Z0-9])` のように直前後の文字種を制限することで、Slackの実際の動作に近づけられる。ただし、このタスクのスコープ定義に「Slackの仕様に厳密に準拠すること」が明示されていない場合、後続タスクで対応するという判断もあり得る。その場合はコメントに既知の制限として明記すること。

2. **[front/src/app/features/mrkdwn/MrkdwnRenderer.tsx:52] `MrkdwnContent`が`div.tweet-contents`ラッパーを持つことへの懸念**: `MrkdwnContent` コンポーネントが `<div className="tweet-contents">` でコンテンツをラップしているが、このコンポーネントは後続タスクで既存表示パイプラインに統合される際に使われる。統合先がすでに `tweet-contents` クラスを持つ要素内にあった場合、ネストが二重になる可能性がある。パーサー+レンダラーのみを実装するこのタスクのスコープでは、`MrkdwnContent` はラッパーなしのフラグメントを返すか、ラッパーを呼び出し側の責務にする設計が望ましい。

3. **[front/src/app/features/mrkdwn/parser.test.ts] `foo_bar_baz` や `2 * 3 * 4` 等の境界ケースのテストが不足**: 上記1番の問題に対応するテストケースが存在しない。Slackでよく登場するスネークケースの変数名やURL中の `~`（例: `https://example.com/~user/`）がどうパースされるべきかのテストを追加することを推奨する。

### 補足（FIX不要）

- `RenderNode` の `switch` 文は `MrkdwnNode` の全ユニオン型をカバーしており、TypeScriptの網羅性チェックが機能している。
