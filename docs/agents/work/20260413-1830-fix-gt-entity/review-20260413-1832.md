### 指摘事項

1. **[front/src/app/features/mrkdwn/parser.ts:104]**: `decodeSlackEntities` での置換順序がセキュリティ上の問題を引き起こす。現在の実装は `&amp;` → `&lt;` → `&gt;` の順で置換しているため、`&amp;gt;` というリテラル文字列が入力された場合に二重デコードが発生する。具体的には `&amp;gt;` → `&gt;` → `>` となり、意図しない `>` が生成される。

   Slack の仕様では `>` は `&gt;` として送信されるが、ユーザーが入力で `&amp;gt;` と書いた場合は Slack が `&amp;amp;gt;` としてエンコードするため、実際のメッセージでこの問題が顕在化するかは用途次第。ただし、正しい実装は `&amp;` を最後に置換するべき（`&lt;` → `<`、`&gt;` → `>`、`&amp;` → `&` の順）。

   修正案:
   ```ts
   function decodeSlackEntities(text: string): string {
     return text.replace(/&lt;/g, '<').replace(/&gt;/g, '>').replace(/&amp;/g, '&');
   }
   ```

2. **[front/src/app/features/mrkdwn/parser.ts:26,54]**: コードブロック（` ``` `）とインラインコード（`` ` ``）内のテキストはエンティティデコードされていない。タスクの修正方針では `<...>` 構文内部のみ言及されているが、コード内にエンティティが含まれる場合の扱いが未定義のまま。

   たとえば `` `if a &gt; b` `` というコードブロックは現在 `{ type: 'code', text: 'if a &gt; b' }` となるが、表示時に `&gt;` をそのまま見せるべきか `>` にデコードすべきかを明確にし、仕様に沿った実装をする必要がある。Slack の公式ドキュメントによればコードブロック内もエンティティエンコードされるため、デコードすべきと考えられる。テストを追加して仕様を明確にすること。
