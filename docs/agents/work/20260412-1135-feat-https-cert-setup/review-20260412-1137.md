### 指摘事項

1. **[front/vite.config.ts:33]**: `.env.local` のパース処理で `line.split('=').map(s => s.trim())` を使っているため、値に `=` が含まれるパス（例: Base64エンコード値やクエリ文字列を含むパス）の場合に2番目以降のセグメントが切り捨てられる。これは既存の問題だが、本PRの変更で `certFile`/`keyFile` がこのパース結果に依存するようになったため、影響範囲が広がっている。修正案: `line.split('=')` の代わりに `line.split(/=(.*)/s)` または `line.indexOf('=')` を使って最初の `=` のみで分割する。

   ```ts
   // 修正例
   .map(line => {
     const idx = line.indexOf('=');
     return [line.slice(0, idx).trim(), line.slice(idx + 1).trim()];
   })
   ```

   ただし、証明書パスに `=` が含まれるケースは実用上ほぼ起きないため、軽微な問題として Dismissed も許容できる。

### Dismissed

- 指摘1: `.env.local` のパースは既存コードの問題であり本PRのスコープ外。証明書パスに `=` が含まれるケースは実用上発生しない。
- 指摘3: 証明書ディレクトリパスにスペースが含まれるケースは実用上発生しない。

2. **[front/vite.config.ts:40-41]**: `certFile` は存在するが `keyFile` が存在しない場合（またはその逆）、警告なしにサイレントで HTTPS が無効化される。例えば `.env.local` に `SSL_CRT_FILE` だけ設定して `SSL_KEY_FILE` を書き忘れた場合、`https` が `undefined` になって HTTP 起動する。ユーザーに意図しない動作をさせないよう、片方だけ設定されている場合はエラーメッセージを出して終了すべき。

   ```ts
   if ((certFile && !keyFile) || (!certFile && keyFile)) {
     console.error('Error: SSL_CRT_FILE and SSL_KEY_FILE must both be set or both be unset.');
     process.exit(1);
   }
   ```

3. **[scripts/setup_pem.sh:10]**: `xargs` を使った値の trim 処理は `.env.local` に引用符なしの値を想定しているが、`.env.local` の値がスペースを含む場合に `xargs` がデフォルトで分割処理を行うため意図しない挙動になりうる。実際の証明書ディレクトリパスにスペースは稀だが、`xargs` の代わりに `tr -d ' '` や変数展開の trim を使う方が安全。現状の一般的な利用では問題ないが、スクリプトの堅牢性の観点から指摘する。

4. **[README.md:42]**: `証明書がなければ \`myinfra/domains/local/\` の手順で発行` と記載されているが、`myinfra/` はこのリポジトリに存在しない（別リポジトリもしくは別ディレクトリを指している可能性がある）。リポジトリ外部のパスを参照する場合、初めて読むユーザーには何を指しているか不明瞭になる。リポジトリパスではなく `（別途管理している myinfra リポジトリの...）` のように補足するか、参照先を明示すべき。
